Data Dictionary

Das DICOM Daten Wörterbuch ist im Wesentlichen ein Register aller in der Medizin verwendeten Standarddatenelemente, also Attribute. Diese Elemente oder Items, werden in die 28 VRs formatiert.

Damit sie in einer Liste mit über 2000 verschiedenen Elementen zuordnungsbar sind, werden sämtliche Elemente in nummerierte Gruppen unterteilt, basierend auf Ähnlichkeiten in Bezug auf deren Inhalt.

Da jede Gruppe nach Elementen organisiert ist, wird jedes Element mit einer eigenen Nummer (Gruppe/Element) bezeichnet, dem sogenannten “Tag”. Diese Bezeichnung wird in Hexadezimalzahlen angegeben.

Das Datenelement «Patientenname» kann durch den Tag (0010,0010) oder den Attributnamen referenziert werden. Dies ist in DICOM von besonderer Bedeutung, da DICOM Anwendungen mit «Tags» auf die Dateielemente verweisen. Sie sind kürzer, besitzen eine feste Grösse und folgen einem hexadezimalen, computerfreundlichen Format.

Für diese Elemente existieren mehrere Bezeichnungen wie etwa «Attribute», «DICOM Datenelemente» oder einfach «DICOM Elemente».

Wie vorgängig erwähnt, sagt die VR aus, in welchem Datenformat eine Information aus der realen Welt in die digitale gelangt. Die Value Multiplicity (VM) definiert hingegen, ob das zugehörige Element nur ein oder mehrere Werte seines Typs enthalten darf. Bei-spielsweise kann das Element (0010,1001) «Andere Patientennamen» durchaus mehr als einen Namen enthalten. Diese Multiplizität wird als «1-n» angegeben und n entspricht dabei einer beliebigen Anzahl. Quelle

Unique Identifier Definition - UID

Die UID wird zur eindeutigen Identifikation für ein Vielzahl von Elementen eingesetzt, wie in der Tabelle oben.
Damit wird die Einzigartigkeit in verschiedenen Ländern, Standorten und Anbietern von Systemen gewährleistet.

Verschiedene Klassen von Objekten, Instanzen von Objekten und Informations Entity IE lassen sich damit im gesamten DICOM Universum voneinander unterscheiden.

Die UID in DICOM basiert auf der OSI-Objektidentifikation (numerische Form) gemäss der Norm ISO 8824.

Jede von DICOM verwendete UID ist gemäss ISO 9834-1 definiert, um die globale Eindeutigkeit sicher zu stellen. Quelle

Value Representation - VR

Im 5. Teil des DICOM Standards werden 28 grundlegende Datentypen (Format) definiert, mit welchen alle möglichen klinischen Daten erfasst werden. Das bedeutet, dass alles was in DICOM geschrieben wird, mit einem dieser 28 VRs übereinstimmen muss.

Jede VR wird beschrieben mit einem abgekürzten Namen aus zwei Buchstaben, einer Definition der jeweiligen VR, einer Beschreibung der zulässigen Zeichen im jeweiligen Datentyp und der vorgeschriebenen Datenlänge. Die Datenlänge oder auch „VR length“ genannt in DICOM, definiert die Datengrösse der DICOM Datentypen. Sie ist ein wesentlicher Bestandteil der VR und für die Entwicklung und Kompatibilität von DICOM Anwendungen von grosser Bedeutung.

Es lässt sich zusammenfassend sagen, dass die VRs die wichtigste Rolle in der DICOM Dateistrukturierung spielen. Sie sind die Wortformen die DICOM versteht sowie spricht und verbinden damit die reale Welt mit DICOM. Quelle

Beispiele

PN VR Der gesamte Personenname (PN) in DICOM wird unter einem eigenen VR abgespeichert. Speziell bei den Namen und deren Reihenfolge können leicht Verwirrung entstehen. Daher gibt der DICOM Standard die Reihenfolge folgend vor: Familienname – Personenname – Mittlerer Name – usw.

UID VR Die UID VR ist eine eindeutige Kodierung von Instanzen bestimmter DICOM Objekte (Daten). Während AE nur lokal wie innerhalb eines Netzwerks eindeutig identifizierbar sind, müssen DICOM UIDs es global sein. So wird beispielsweise bei einer Kopie einer Studie eine neue UID vergeben um Verwechslungen nach Bearbeitung der Studien zu vermeiden.

AE VR Die DICOM AE VR repräsentiert die DICOM Application Entity oder anderst gesagt, eine DICOM Anwendungseinheit. Das kann ein DICOM Gerät oder Software sein und dient zur eindeutigen Identifikation in einem Netzwerk mittels dem AE – Title, oder einfach dem Titel (Name), der dem Gerät oder Software zugewiesen wurde. Es können keine identischen AE Titel in einem Netzwerk existieren, was diese VR zu einer der wichtigsten für DICOM Netzwerke und PACS macht.

Hier gilt es, die Wichtigkeit von der Wahl des Titels zu erwähnen. Gemeint ist damit, dass auf die Wahl eines Titels mit Gross und Kleinschreibung verzichtet werden soll. Damit lassen sich unnötige Kommunikationsprobleme vermeiden, da dies bei einigen DICOM Geräten zu Problemen führt.

Zahlen in Binär VRs Es handelt sich hierbei um die gleichen Zahlen wie bei (IS) und (DS) die jedoch binär abgespeichert werden. Signed Short (SS), Unsigned Short (US), Signed Long (SL), Unsigned Long (UL), Floating Point Single (FL) und Floating Point Double (FD) werden verwendet um einzelne Zahlen darzustellen. Other Byte String (OB), Other Word String (OW) und Other Floating String (OF) werden für lange numerische Zeichenketten verwendet. Mit dem Attribute Tag (AT) werden ausschliesslich DICOM Attribute beschrieben. Dieser Datentyp kennzeichnet jedes Attribut mit einem Gruppen- und Element Tag. Darauf wird später noch näher eingegangen.

Zahlen in Text VRs Um nummerische Werte als Textzeichenfolge abzuspeichern, werden die Datentypen Integer String (IS) und Decimal String (DS) verwendet. Trotz der schlechteren Eignung als Binäre Daten im Vergleich, sind die textformatierten Zahlen in DICOM häufig anzutreffen. Einerseits weil sie leicht leserlich sind und daher auf vielen Oberflächen angezeigt werden können, anderseits weil sie unabhängig von Little und Big Endian Bytereihenfolgen sind.

Datum & Zeit VRs In DICOM gibt es ganze 4 VRs um die Zeit und das Datum in einem String, also Text, zu speichern. Date (DA), Time (TM), Date Time (DT) und Age String (AS). Wichtig hierbei ist vor allem die Reihenfolge der einzelnen Angaben wichtig. Diese hat in HHMMSS zu erfolgen und wäre beispielsweise für 18:30:00 in DICOM als 183000 geschrieben.

Die Anzahl der VRs für Datum & Zeit ist ein Beispiel dafür wie vielfältig DICOM aufgebaut ist und zum Teil auch eher unnötige Aufteilungen beinhalten kann. Quelle

Module

Module oder Informationsmodule sind die erste Ebene in der Organisation der DICOM Daten. Ihr Hauptzweck besteht darin, verwandte Attribute (Datenelemente) in eine sinn-volle Reihenfolge zu gliedern. Die richtige Auswahl eines Moduls hängt dabei von der Art der gespeicherten Daten im DICOM Objekt ab.

Wenn man alle Informationen zur Identifikation eines Patienten gruppiert, ergibt sich daraus das Patienteninformationsmodul mit dem Inhalt; Patientenname, Grösse, Gewicht, etc. In diesen Informationsmodulen müssen jedoch nicht alle Werte belegt sein. Zur Kennzeichnung werden diese mit Grossbuchstaben, wie M für obligatorisch, C für bedingte und U für Benutzerdefinierte Informationsmodule, beschrieben.

Es gibt beispielsweise Module für Geräte (Modalitäten), EKGs, Berichte oder Sprachaufzeichnungen. Ausserdem ist das Patientenidentifikationsmodul für Modalitäten Pflicht. Das heisst, es kann kein Bild existieren, welches nicht einem Patienten zugeordnet werden kann.

In Teil 3 des DICOM Standards sind Standardmodule und Informationsobjektmodule definiert in welchen angegeben wird, welche Module für die jeweiligen Modalitäten geeignet sind.

Informatiion Entities

Auf der nächsten Ebene der Organisation der DICOM Daten sind die Information Entities. Nachdem die Attribute gruppiert wurden, werden diese durch Information Entities als reale Einheiten dargestellt, welche am Workflow der medizinischen Bildgebung beteiligt sind. Dabei ist eine Information Entity eine Auflistung von relevanten Modulen die darin enthalten sein sollten.

So müsste beim allgemeinen Patienten IE das Patientenmodul, das Probeidentifikationsmodul und das Subjektmodul für klinische Studien enthalten sein. Oder die IE für allgemeine Studien umfasst das allgemeine Studienmodul, das Patientenstudienmodul und das klinische Studienmodul. IE’s für Bilder und Serien sind wesentlich komplexer aufgebaut, da mehr Module beteiligt sind. Anhand des DICOM Informationsmodells mit dem hierarchischen Workflow lässt sich erkennen, was damit gemeint ist. Dabei ent-sprechen die Rechtecke den IE’s.

IOD

An der Spitze der Organisation von DICOM Daten befinden sich die IOD’s. Eine sinnvolle Kombination aus IE’s ergibt das jeweilige IOD, mit welchen, die von DICOM verwendeten Objekte definiert sind. Gemeint ist damit, dass die gesamte Datenverarbeitung über IODs läuft für DICOM Anwendungen und Modalitäten. Mit IOD’s werden die häufigsten Datentypen in der medizinischen Bildgebung erstellt, wie Bilder für die verschiedenen Modalitäten.

Unterschieden werden generell zwei Arten von IOD’s. Zum einen die normalized IOD und zu anderen die composite IOD. Die normalized IOD beinhalten nur Attribute aus einer Gruppe. Das heisst eine reine Patienteninformation wäre normalized, da nur Attribute aus der Gruppe der Patientendaten enthalten sind. Weitere Beispiele wären die Studieninformation oder die Serieninformation. Die composite IOD beinhalten Attribute aus verschiedenen Gruppen.

Bei der Klasse der CT-Bilder beispielsweise sind nicht nur Informationen zum Patienten vorhanden, sondern auch Informationen aus den Gruppen der Serie und Studie

In DICOM IOD’s werden verschiedene Attribute der unterschiedlichen Gruppen zu Objektklassen zusammengefasst. Die Klasse der CT-Bilder ist ein Beispiel davon, zu welcher bestimmte Attribute angehören.
