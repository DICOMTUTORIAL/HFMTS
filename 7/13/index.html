Verification

Die Verification Class dient der überprüfung einer DICOM Konnektivität zwischen 2 DICOM Knoten. Es wird als SCU und SCP eingesetzt. Der Verlauf beginnt mit einer C-ECHO-RQ Message vom SCU zum SCP, der wiederum mit einer C-ECHO-RSP antwortet. Damit wird sichergestellt, das diese Verbindung zwischen den DICOM Knoten richtig konfiguriert ist.

Die Verification Class lässt sich wie folgt ausführen:

PC

  1. Nach der Wahl der DICOM Servers.
  2. Wird mit der Taste Test ECHO das C-ECHO SCU ausgeführt.
  3. Nach erfolgter Antwort des SCP erscheint das Echo result.

Workstation

  1. Mit der Taste Send to C-Store öffnet sich.
  2. Das Fenster für die Auswahl des SCP mit der Taste ECHO erscheint.
  3. Ein weiteres Fenster erscheint mit dem Status der Verification Class.

RPI

Der gleiche Vorgang in Orthanc auf dem Raspberry Pi via ssh. Zusätzlich wird -d angehängt für mehr Informationen mit der debug option.

$   echoscu 192.168.1.20 4444 -d

Am Ende ist das Resultat des Verlaufs ersichtlich. I: Received Echo Response (Success)

Konzept

Die Verification SOP dient der Überprüfung einer erfolgreichen DICOM Konnektivität zwischen zwei AEs. Dazu verwendet die Verification SOP die DIMSE Nachricht C-ECHO. Da zur Überprüfung der Konnektivität keine Datenverarbeitung erforderlich ist, besteht das einfachste und doch sehr häufig genutzte Service Object Pair nur aus der DIMSE Nachricht, also ohne DICOM Objekt (IOD). Dazu wird vom Verifizierungs SCU eine Verifikationsanforderung gesendet, genauer gesagt eine C-ECHO-Rq an eine andere DICOM AE. Nur wenn die andere DICOM AE ordnungsgemäss angeschlossen ist und die Verification SOP als SCP unterstützt, wird eine Antwort als C-ECHO-Rsp übermittelt um die funktionierende Verbindung zu bestätigen. (Pianykh, 2008, S. 131-132)
